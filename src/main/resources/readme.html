Federation Plugin

This plugin lets multi‑user chat (MUC) rooms span several Openfire servers — even servers that are not directly connected to each other. Once two rooms are linked, their occupants and messages merge into a single conversation. Your users do nothing special: they join their normal local room and automatically see everyone across the federation.

Before you start

  • Install this plugin on every server that should take part.
  • Each pair of neighbouring servers needs working XMPP server‑to‑server (S2S): resolvable domains (DNS/SRV), open ports, and matching TLS settings. Adding a peer here opens the S2S connection for you.
  • Servers do not all need to connect directly — the plugin relays traffic multi‑hop along the path it learns.

The Federation tab

After the plugin loads, a Federation tab appears in the admin console with three views. The page auto‑refreshes every 5 seconds.

ViewWhat it does
Peer Servers Add, remove or disable peer servers; see their status and last‑seen time; watch live S2S sessions; and tune connection settings (keepalive & reconnect).
Routing Table Shows every destination the server can reach, the next hop toward it, and the hop count. A hop count of 1 means a direct S2S link. Routes are learned automatically.
Rooms Turn federation on or off per local room, see and manage room mappings, and browse the rooms your peers have advertised.

Step‑by‑step: federate a room

1. Add your peers. On the Peer Servers tab, under Add peer server, enter the other server's XMPP domain (for example chat.example.com) and click Add peer. Do this on each server for its neighbours. Watch the status dot: reachable, connecting/retrying, unreachable, disabled.
2. Turn on federation for a room. On the Rooms tab, find the room under Local rooms and switch its Federated toggle on. The room is now advertised to your peers.
3. Link the rooms. On a peer server, the advertised room now appears under Remote rooms. Map your local room to it (or, from the other side, map theirs to yours). Once mapped, the two rooms become one shared conversation. A mapping shows connected when the route is up and disconnected when it is lost.
4. Done. Users join their local room as normal and see occupants and messages from every mapped server. Repeat for as many servers as you like; with multi‑hop, a room can chain across servers that never talk directly.
Tip: federate the same room name on several servers and map them in a hub‑and‑spoke or chained layout. Presence, typing, and message history forwarding all follow the mappings.

Connection settings

On the Peer Servers tab, the Connection settings box exposes:

SettingDefaultMeaning
S2S keepalive interval240 s How often a lightweight ping is sent to each reachable peer to detect liveness quickly. Minimum 30 s.
Max reconnect interval (back‑off cap)30 s Upper bound for retrying an unreachable peer. Retries grow 5 → 10 → 20 s … up to this cap, then reset once the peer reconnects. Minimum 5 s.

Idle connections

The plugin manages S2S liveness itself, so on startup it disables Openfire's built‑in S2S idle timeout (xmpp.server.idle). This prevents repeated “Connection has been idle” reconnects that would otherwise occur because XMPP uses a separate one‑way socket per direction. This change is server‑wide — it affects all S2S connections, not just federation peers.

To keep Openfire's idle timeout in place instead, set the system property plugin.federation.disableS2SIdle to false (Admin Console → Server → System Properties).

Security

  • Rooms are opt‑in. A remote peer can only map, or inject presence/messages into, a local room you have explicitly marked Federated. Forwarded traffic aimed at any other local room is dropped (logged with a SECURITY: tag).
  • Peer allowlist (default on). plugin.federation.peerAllowlist defaults to true: only peers you Add here may federate; all others are rejected. Both ends must add each other (auto-registration of unknown peers is suppressed). Set it to false (or use the Security toggle on this tab) for open federation, accepting any server that connects.
  • Untrusted peers (filtered exposure). Tick Untrusted when adding a peer (or use Make untrusted on its row) and it gets no routing updates and no room advertisements. Open its Rooms editor to choose exactly which rooms it may see — from your federated local rooms and any remote rooms reachable through this server. It is sent only those rooms and only routes to the servers hosting them, and any mapping/message it sends toward a room you did not expose is dropped (logged SECURITY:). Use this for an edge server fronting a partner organisation that should see only a curated set of rooms.
  • Admin API CSRF. This tab's actions are protected by a double‑submit token. After upgrading the plugin, reload an already‑open Federation tab once so it picks up the new token.

Managing peers

  • Remove — soft removal; tears down the link and clears mappings to that peer. The peer can be added again later.
  • Disable — an authoritative, persistent block. The disabled peer cannot re‑establish the connection (even by re‑adding) until you re‑enable it.
  • Untrusted — toggle filtered exposure for a peer (see Security). An untrusted peer's Rooms editor lets you pick precisely which rooms it may see; everything else in your network stays hidden from it.

Troubleshooting

SymptomWhere to look
Peer stays unreachable Check DNS/SRV for the peer domain, firewall/ports, and TLS. The Routing Table won't list a peer you can't reach directly or via a hop.
Remote room doesn't appear Confirm the room's Federated toggle is on at the origin, and that a route to that server exists in the Routing Table.
Users don't see each other Confirm the rooms are mapped (not just federated) and the mapping shows connected.
Mapping shows disconnected The route to the remote server is down — check the peer status and the path in the Routing Table.

Federation Plugin

This plugin lets multi‑user chat (MUC) rooms span several Openfire servers — even servers that are not directly connected to each other. Once two rooms are linked, their occupants and messages merge into a single conversation. Your users do nothing special: they join their normal local room and automatically see everyone across the federation.

Before you start

  • Install this plugin on every server that should take part.
  • Each pair of neighbouring servers needs working XMPP server‑to‑server (S2S): resolvable domains (DNS/SRV), open ports, and matching TLS settings. Adding a peer here opens the S2S connection for you.
  • Servers do not all need to connect directly — the plugin relays traffic multi‑hop along the path it learns.

The Federation tab

After the plugin loads, a Federation tab appears in the admin console with three views. The page auto‑refreshes every 5 seconds.

ViewWhat it does
Peer Servers Add, remove or disable peer servers; see their status and last‑seen time; watch live S2S sessions; and tune connection settings (keepalive & reconnect).
Routing Table Shows every destination the server can reach, the next hop toward it, and the hop count. A hop count of 1 means a direct S2S link. Routes are learned automatically.
Rooms Turn federation on or off per local room, see and manage room mappings, and browse the rooms your peers have advertised.

Step‑by‑step: federate a room

1. Add your peers. On the Peer Servers tab, under Add peer server, enter the other server's XMPP domain (for example chat.example.com) and click Add peer. Do this on each server for its neighbours. Watch the status dot: reachable, pending / connecting / retrying, unreachable, disabled. A peer stays Pending until the other side adds this server back — federation only starts once both ends have added each other. A server already talking plain S2S to you appears under Active S2S sessions with an Add peer button, so you can federate it in one click.
2. Turn on federation for a room. On the Rooms tab, find the room under Local rooms and switch its Federated toggle on. The room is now advertised to your peers.
3. Link the rooms. On a peer server, the advertised room now appears under Remote rooms. Map your local room to it (or, from the other side, map theirs to yours). Once mapped, the two rooms become one shared conversation. A mapping shows connected when the route is up and disconnected when it is lost.
4. Done. Users join their local room as normal and see occupants and messages from every mapped server. Repeat for as many servers as you like; with multi‑hop, a room can chain across servers that never talk directly.
Tip: federate the same room name on several servers and map them in a hub‑and‑spoke or chained layout. Presence, typing, and message history forwarding all follow the mappings.

Direct (1:1) federation

Beyond rooms, the overlay relays one‑to‑one traffic to contacts on servers you can only reach through a hop. 1:1 chat, presence (online/away/dnd) and subscriptions all traverse multi‑hop links. As of 1.6.0 this extends to profile data: a contact's vCard and avatar (XEP‑0054/0153) now load across the overlay. The vCard reply is built at the contact's home server and relayed back to the requester. Forwarded presence carries the avatar hash and entity‑ caps node, so clients know there is an avatar to fetch and what the contact supports. IQs answered by the contact's client (e.g. disco/version to a specific resource) also round‑trip; server‑ answered non‑vCard IQs are best‑effort.

Directly‑linked (adjacent) peers keep using native server‑to‑server for all of the above — relay only kicks in where there is no direct link. Jingle audio/video and file transfer are not relayed; they need a separate media path and are out of scope.

Users tab & bookmark advertising

The Users tab lists this server's currently‑connected clients and, grouped by peer, the online users other servers have published through the directory gossip. As of 1.7.0 it can also advertise your connected clients to peers as XEP‑0048 bookmarks: each connected client is pushed across the overlay and injected into each peer user's bookmark storage as a <url> bookmark (xmpp:user@server?message), so it shows up in a normal chat client. Use Push bookmarks to peers for a one‑shot, or enable Auto‑push to re‑advertise automatically as users come and go. Only trusted peers receive it; the injected entries are tagged so they merge with — never overwrite — a user's own bookmarks, and turning auto‑push off withdraws them. Bookmarks are read by clients at login, so changes appear on a peer user's next sign‑in.

Connection settings

On the Peer Servers tab, the Connection settings box exposes:

SettingDefaultMeaning
S2S keepalive interval240 s How often a lightweight ping is sent to each reachable peer to detect liveness quickly. Minimum 30 s.
Max reconnect interval (back‑off cap)30 s Upper bound for retrying an unreachable peer. Retries grow 5 → 10 → 20 s … up to this cap, then reset once the peer reconnects. Minimum 5 s.

Idle connections

The plugin manages S2S liveness itself, so on startup it disables Openfire's built‑in S2S idle timeout (xmpp.server.idle). This prevents repeated “Connection has been idle” reconnects that would otherwise occur because XMPP uses a separate one‑way socket per direction. This change is server‑wide — it affects all S2S connections, not just federation peers.

To keep Openfire's idle timeout in place instead, set the system property plugin.federation.disableS2SIdle to false (Admin Console → Server → System Properties).

Security

  • Rooms are opt‑in. A remote peer can only map, or inject presence/messages into, a local room you have explicitly marked Federated. Forwarded traffic aimed at any other local room is dropped (logged with a SECURITY: tag).
  • Peer allowlist (default on). plugin.federation.peerAllowlist defaults to true: only peers you Add here may federate; all others are rejected. Both ends must add each other (auto-registration of unknown peers is suppressed). Set it to false (or use the Security toggle on this tab) for open federation, accepting any server that connects.
  • Untrusted peers (filtered exposure). Tick Untrusted when adding a peer (or use Make untrusted in its settings panel) and it gets no routing updates and no room advertisements. Open its settings panel (the expand arrow on its row) to choose exactly which servers it may see — this server itself (its federated local rooms) and any server reachable through this one. It is sent only the federated rooms homed on those servers, plus a route to each, and any mapping/message it sends toward a room homed on a server you did not expose is dropped (logged SECURITY:). Use this for an edge server fronting a partner organisation that should see only a curated set of servers.
    Trust is per‑link: both ends must set the same trust level. If one side marks the link untrusted and the other leaves it trusted, the link shows Trust mismatch and carries no federation until they agree — then it comes up on its own. The settings panel shows both directions: the servers you expose to that peer (editable), and the servers that peer is advertising through to you — each with a Deny button (see below).
    Untrusted by default: adding a peer under a different parent domain (last two DNS labels, e.g. example.net; adjust with plugin.federation.trustDomainLabels) ticks Untrusted automatically; same‑parent peers default trusted.
  • Deny a route advertisement (per‑link). Exposure controls what you send; each side can also refuse what it receives. If a peer advertises a route to a server you don't want, click Deny — on the Routing Table row, or next to that server in the peer's settings panel. The destination is refused whenever that peer advertises it: any installed route via that peer is removed at once (rooms and ghost occupants cleaned up) and later advertisements are dropped. The entry stays in the Routing Table as a disabled row and the deny is remembered even if the route is withdrawn and advertised again later. A route to the same destination via a different peer is unaffected, and nothing is negotiated with the peer — the deny is entirely yours. Allow lifts it — on the disabled row, or in the peer's settings panel. Servers on the far side of a deny (whose own routing table still shows a route) detect the break end‑to‑end: active mappings are probed with a mapping-ping every 30 s; when the pongs stop (≈2 minutes), the mapping shows ⚠ not responding and its remote occupants are dropped, then everything re‑syncs on its own once the path answers again. Which domains answer probes is remembered persistently, so a path that is already broken when the plugin starts is still detected; the room's settings panel shows each mapping's probe round trip and last‑answer age (ping 12 ms · 3s ago).
  • S2S key pinning (trust‑on‑first‑use). The first time a peer's S2S link comes up, the SHA‑256 of the leaf certificate's public key (SPKI) is pinned. If the key later changes (a server re‑created under the same domain name — even with a cert from the same CA), the peer is auto‑marked untrusted and flagged ⚠ cert changed; federation toward it is blocked until you click Trust new cert. Renewals that reuse the key pass silently; a key rotation raises the flag. Needs a TLS‑secured S2S link — a plain dialback link presents no certificate and is not pinned. Pre‑1.7.13 pins (top‑of‑chain cert hashes) upgrade in place on the next sighting.
  • Sender‑identity (anti‑spoofing) checks. Every stanza forwarded over the overlay has its claimed from validated at each hop: no peer may deliver a stanza pretending to come from this server's own users, and an untrusted peer may only speak for itself or for servers reached through it — a forged identity from the wrong direction is dropped and logged with a SECURITY: tag.
  • Per‑room visibility. Each federated room has a Visible control listing the servers allowed to see it — pick from the routable peers, or add a server manually before it is reachable (the room advertises to it automatically once a route appears). A newly‑federated room defaults to visible to nobody; pick servers, or tick Visible to all peers. The visibility set travels with the advertisement so every relay confines the room to the path toward its allowed destinations; the ACL is persisted and survives a listed server being offline. (On‑path servers must run this version or newer; rooms already federated at upgrade are migrated to visible to all so existing federation keeps working.)
  • Mapping consent. Mapping onto a remote room sends a request the other admin must Accept (a Pending mapping requests panel + an inline prompt on the room) before any traffic flows; Reject declines it. On accept a per‑mapping token is shared and re‑checked on later lifecycle messages, so a remove+re‑add forces a fresh acceptance. A mapping can be Disabled instead of removed (the peer shows "disabled by peer") and re‑enabled. A per‑room Auto‑accept toggle accepts incoming requests automatically. On upgrade, existing mappings drop to pending and must be re‑accepted (the lower‑domain side auto‑re‑requests on reconnect).
  • Admin API CSRF. This tab's actions are protected by a double‑submit token. After upgrading the plugin, reload an already‑open Federation tab once so it picks up the new token.

Managing peers

  • Remove — soft removal; tears down the link and clears mappings to that peer. The peer can be added again later.
  • Disable — an authoritative, persistent block. The disabled peer cannot re‑establish the connection (even by re‑adding) until you re‑enable it.
  • Untrusted — toggle filtered exposure for a peer (see Security). An untrusted peer's settings panel lets you pick precisely which servers it may see; everything else in your network stays hidden from it.

Troubleshooting

SymptomWhere to look
Peer stays unreachable Check DNS/SRV for the peer domain, firewall/ports, and TLS. The Routing Table won't list a peer you can't reach directly or via a hop.
Remote room doesn't appear Confirm the room's Federated toggle is on at the origin, and that a route to that server exists in the Routing Table.
Users don't see each other Confirm the rooms are mapped (not just federated) and the mapping shows connected.
Mapping shows disconnected The route to the remote server is down — check the peer status and the path in the Routing Table.

Connection settings
S2S keepalive interval
Sends a lightweight ping to each reachable peer at this interval to prevent Openfire from closing idle S2S sessions. Must be less than Openfire's xmpp.server.idle setting (milliseconds ÷ 1000 × 0.8). Minimum: 30 s. Default: 240 s — safe for Openfire's built-in 6-minute idle timeout.
s
Max reconnect interval (back-off cap)
Maximum interval between reconnection attempts for UNREACHABLE peers. Retries use exponential back-off starting at 5 s: 5 s → 10 s → 20 s → … capped at this value. On reconnect the back-off resets automatically. Minimum: 5 s. Default: 30 s.
s
Mapping probe interval (mapping-ping)
Probes every active room mapping end-to-end at this interval; the mapped domain answers with a pong routed back across the overlay. Three unanswered probes in a row flag the mapping as ⚠ not responding and evict its remote occupants (≈2 minutes at the default cadence). Applies immediately — no restart needed. 0 disables the probe. Minimum: 15 s. Default: 30 s.
s
Security & presence
Restrict federation to approved peers
When enabled, only peers you have explicitly added here may federate; federation actions from any other server are rejected. Disable for open federation (accept any server that can connect). Existing peers stay approved when this is turned on.
…
Allow users to reach remote rooms across federation
When enabled (default), a user on another server may traverse the federation network to join or post to a local MUC room directly over S2S, even when no federation mapping exists — so ad-hoc and private rooms that are never advertised stay reachable. Federation's own traffic is unaffected. Disable to lock rooms down so they are only reachable through an explicit federation mapping.
…
Relay 1:1 messages over federation
When enabled, a private message a local user sends to a user on an overlay-reachable peer is relayed hop-by-hop through federation (so it works even across untrusted edges with no direct S2S). Replies route back the same way. Disable to use Openfire's native S2S for 1:1 chat.
…
Probe federated contacts on subscription
When a local user adds a contact on a multi-hop peer, Openfire's own presence probe leaks to native S2S and fails (no direct link), so the contact never shows online. With this on, the probe is re-issued over the federation overlay so the contact's presence is pulled. Recommended on.

Publish my user directory to peers
When enabled, this server gossips its list of currently-online users so peers can see who is reachable here (see the Users tab). Off by default for privacy; untrusted peers never receive it. Messaging by typed JID still works when this is off.

Default settings for new rooms (by name pattern)

When a room is created, the matching rule below is applied automatically. Patterns use shell-style globs — * matches any text, ? one character (e.g. *_ext matches mychat_ext; * matches every room). When several rules match a name the most specific one wins (most literal characters), so * acts as a catch-all. A non-federated rule excludes a name pattern from a broader rule. Auto-map requests a same-named room link on each visible peer; combined with the peer's auto-accept, two servers using the same naming rule auto-link with no clicks.

Advertise connected clients as bookmarks (XEP-0048)
Auto-push my connected clients to peers
When enabled, this server's currently-connected clients are advertised to every trusted peer and injected into each peer user's bookmark storage as XEP-0048 <url> bookmarks — so they appear in a normal chat client. Re-pushes automatically as users come and go. Untrusted peers never receive it. Bookmarks are read at login, so a peer user sees changes on their next sign-in. Turning this off withdraws the bookmarks this server pushed.

File sharing
Federate file shares (HTTP File Upload)
When enabled (default), a file shared via HTTP File Upload is relayed over the federation to exactly the servers whose users receive the message (mapped-room peers, a 1:1 recipient's home — never broadcast), re-hosted there, and the link is rewritten so every client downloads from its own server. Local-only traffic never leaves; transit hops never store a copy.
…
Maximum file size
Largest file the relay will stage, transfer, or accept from a peer. Files above the cap keep their original (origin-server) link. Minimum: 1 MB. Default: 25 MB.
MB
File retention
How long a relayed file stays in this server's federation-files store (and its rewritten link keeps working) before it is purged. Minimum: 1 day. Default: 90 days.
days
Storage directory
Full path of the folder where relayed file content is kept on disk. Default: /var/lib/openfire/federation-files. Saving moves the existing files to the new location.
Allowed file extensions
Comma-separated list of extensions the relay will stage (egress, from a local upload) or accept (ingress, from a peer). Applies only at the origin and destination — a transit hop relays without ever decoding content, so this never affects traffic just passing through. Blank allows nothing; * allows everything.
Virus scan received files (ClamAV)
When enabled, a file's content is scanned by a clamd daemon before it becomes servable to a local recipient (destination only — never a transit hop). A scan that cannot complete (clamd unreachable) is treated as a failure: the relay fails closed rather than serving unscanned content while this is on.
…
ClamAV endpoint
Host and port of the clamd INSTREAM endpoint. Default clamav:3310 matches the sidecar container added to this server's docker-compose.yml, reachable on the same bridge network.
